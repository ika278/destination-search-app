# 技術選定

## フロントエンド

### React 18 + TypeScript (strict) + Vite
- ReactはUIライブラリとして最もシェアが高く、学習リソースが豊富
- TypeScript strictモードで型安全性を担保
- ViteはHMRが高速で開発体験が良い

---

## APIサーバー

### Hono（Node.js）
- マルチランタイム対応（Node.js / Cloudflare Workers / Bun）でポータビリティが高い
  - Cloudflare Workersが相性が良いが、開発者の経験的にNode.jsを採用した
- TypeScriptファーストで型安全なRPCが使える
- 軽量でシンプル

### AWS Lambda Web Adapter
- Honoアプリを通常のNode.jsサーバーとして書くだけでLambda上で動く
- コードに手を加えず将来ECSなどへ移行できる
- Hono専用のLambda Adapterより移植性が高い

---

## ORM

### Prisma
- Node.js環境では制限がなく、DX・学習コストが最も優秀
- 自動生成される型定義でTypeScriptとの相性が良い
- マイグレーション機能が充実
- Cloudflare WorkersではなくNode.jsで動かすと決めたため、Drizzleより優先した

---

## データベース

### PostgreSQL（AWS RDS）
- MySQL・PostgreSQLはRDSで料金が同じため、コスト面での差はなし
- 位置情報拡張（PostGIS）が使えるため、移動時間・距離計算の将来的な活用に有利
- 学習目的としてもPostgreSQLの方が応用範囲が広い（分析系・クラウドネイティブ系で主流）

---

## インフラ（AWS）

### S3 + CloudFront（フロントエンド）
- 静的サイトホスティングの標準構成
- CloudFrontでCDN配信

### Lambda（APIサーバー）
- サーバー管理不要でコストが低い
- AWS Lambda Web Adapterを使うことでHonoをそのまま動かせる
- 将来的にECS Fargateへの移行も容易

### RDS PostgreSQL（DB）
- マネージドDBでバックアップ・パッチ管理が不要
- 学習目的でAWSを選定

---

## 外部API（地図・移動時間）

### Google Maps Platform
- ジオコーディング（スポット名/住所 → 緯度経度）: Geocoding API
- 移動時間計算（現在地 → スポット）: Routes API
- 毎月$200クレジットが無料付与されるため、MVPフェーズでは実質無料
- 日本語対応・精度・情報量で最も信頼性が高い
- 移動時間はリアルタイムで計算（現在地が検索時まで不明なため、事前計算は不可）
- 無料枠（月$200クレジット）を超える場合に備え、Google Cloud Consoleで予算アラートを設定する

---

## 認証

### Amazon Cognito
- AWS構成との親和性が高く、学習目的でAWSを選定している方針と一致
- メール + パスワード認証を採用（OAuthは不要と判断）
- CognitoはJWTベース（IDトークン + アクセストークン）で、セッションをサーバー側で管理しない
- トークンはhttpOnly CookieにJWTを保存する方式を採用（XSS対策のため）
- Clerk等の認証SaaSはシンプルだが、AWS外部のサービスに依存するため不採用

---

## テスト

- Viteと統合されており設定が最小限
- Testing LibraryでReactコンポーネントをユーザー視点でテスト

### Playwright（E2E）
- モダンなE2Eテストの標準
- クロスブラウザ対応
